기획서 파일 `$ARGUMENTS`를 Read 툴로 읽은 후 아래 6단계 파이프라인을 순서대로 실행하세요.
각 단계 제목을 출력하며 진행 상황을 표시하세요.

---

## [1단계] 구조 분석

`$ARGUMENTS` 파일을 읽어 아래 항목을 파악하고 표로 출력하세요.

1. `##` 헤더 기준으로 기능 섹션 목록 추출
2. 각 섹션에 아래 항목이 있는지 확인 (없으면 ❌ 표시)
   - 화면 상태 정의 (정상/로딩/에러/빈 상태)
   - 수치·정책 기준 (확정값 명시 여부)
   - 수용 기준 AC (Given/When/Then)

출력 형식:
| 섹션명 | 상태 정의 | 수치 기준 | AC |
|--------|----------|----------|-----|
| ...    | ✅/❌    | ✅/❌    | ✅/❌ |

---

## [2단계] 케이스 설계

각 섹션에 대해 아래 기준으로 테스트 케이스를 설계하세요.

**케이스 유형 기준:**
- **해피**: 모든 조건이 정상일 때 핵심 기능이 동작하는 시나리오 (섹션당 최소 1개)
- **언해피**: 잘못된 입력·필수값 누락·유효성 검사 실패 시나리오 (섹션당 최소 1개)
- **엣지**: 아래 4가지 관점에서 각 섹션에 해당하는 항목 도출 (섹션당 최소 2개)
  - 빈 상태: 데이터 0건, 목록 비어 있을 때
  - 권한/인증: OS 권한 거부, 세션 만료, 인증 실패
  - 네트워크: 연결 차단, 타임아웃, 지연
  - 경계값: 최대/최소값, 한도 초과, 형식 불일치

기획서에 AC(Given/When/Then)가 있으면 해당 AC를 테스트 케이스로 직접 변환하세요.

**기대 결과 작성 규칙 (필수):**
- 팝업 노출, 토스트 메시지, 버튼 비활성화, 화면 이동 등 실제 UI 액션으로 작성
- "정책에 따라", "일관 적용", "~또는~" 같은 모호한 표현 사용 금지
- 구체적인 UI 문구가 기획서에 있으면 그대로 인용

---

## [3단계] 우선순위 자동 분류

아래 키워드 룩업 테이블을 기준으로 각 케이스의 우선순위를 분류하세요.

**P0 — 치명적 (서비스 중단·데이터 유실·핵심 기능 불가):**
- 앱 크래시, 강제 종료, 튕김
- 로그인·회원가입 완전 불가
- AI 핵심 기능(캐릭터 생성·일기 생성) 완전 실패
- 데이터 유실, 정합성 손상, 원자성 보장 실패
- 결제·인증 완전 차단
- 핵심 CRUD(생성·조회·수정·삭제) 전체 불가

**P1 — 주요함 (예외 처리 누락·유저 플로우 단절):**
- 예외 처리 미구현, 오류 메시지 미노출
- 유저 플로우 단절, 이전 화면 복귀 불가
- 네트워크 오류·타임아웃 미처리
- OS 권한 거부 미처리
- 경계값(한도 초과·최소값 미만) 오류
- 알림 미수신, 설정 미반영
- 선행 조건 미충족 시 안내 없음
- 계정·인증 관련 예외 (링크 만료, 계정 충돌)

**P2 — 경미함 (UI/UX·빈 상태·디자인):**
- 빈 상태(Empty State) 화면 미정의
- 로딩 상태 표시 미구현
- 디자인 가이드 미준수
- 단순 문구·레이아웃 어색함

## [4단계] 엣지케이스 보완

2단계에서 설계한 케이스를 아래 체크리스트로 검토하여 누락 항목을 'AI추가'로 보완하세요.

**모든 섹션 공통 체크:**
- [ ] Empty State 케이스가 있는가?
- [ ] 네트워크 차단·타임아웃 케이스가 있는가?
- [ ] 경계값(최대·최소·한도 초과) 케이스가 있는가?
- [ ] 선행 조건 미충족 케이스가 있는가?

**파일 업로드 관련 섹션 추가 체크:**
- [ ] OS 카메라·갤러리 권한 거부 케이스가 있는가?
- [ ] 파일 용량 초과 케이스가 있는가?
- [ ] 지원하지 않는 파일 형식 케이스가 있는가?

**AI 연산 관련 섹션 추가 체크:**
- [ ] AI 서버 완전 실패(타임아웃) 케이스가 있는가?
- [ ] 연산 중 앱 강제 종료 케이스가 있는가?
- [ ] 일일 한도 초과 케이스가 있는가?

**계정·인증 관련 섹션 추가 체크:**
- [ ] 인증 링크 만료 케이스가 있는가?
- [ ] 소셜·이메일 계정 충돌 케이스가 있는가?
- [ ] 삭제 후 재가입 케이스가 있는가?

## [5단계] CSV 생성

아래 9개 컬럼을 파이프(|) 구분자로 하나의 코드 블록에 출력하세요.

**컬럼 순서:**
`ID | 섹션 | 케이스 유형 | 테스트 시나리오 | 기대 결과 | 우선순위 | 출처 | 사전 조건 | 비고`

**ID 부여 규칙:** TC-001부터 섹션 순서대로 부여
**케이스 유형:** 해피 / 언해피 / 엣지 중 하나
**출처:** 기획서 AC 기반 → '기획서기반' / 엣지 보완 추가 → 'AI추가'
**비고:** 기획서에 정책이 명시되지 않은 항목, 기획 누락 의심 포인트 기재

CSV 출력 후 아래 명령으로 파일을 저장하세요.
- 파일명: 기획서 파일명에서 `.md` 제거 후 `_tc_YYYYMMDD.csv` 추가
  (예: `product_spec_petbbi_v2_tc_20260701.csv`)
- 저장 위치: `$ARGUMENTS`와 같은 폴더
- 인코딩: UTF-8 BOM (엑셀 한글 지원을 위해 PowerShell로 저장)

PowerShell 저장 명령 예시:
```powershell
$content = @"
[CSV 내용]
"@
[System.IO.File]::WriteAllText("[저장경로]", $content, [System.Text.Encoding]::UTF8)
```

## [6단계] 커버리지 리포트 출력

CSV 저장 완료 후 아래 형식으로 커버리지 리포트를 출력하세요.

```
==============================================================
  QA 테스트 케이스 커버리지 리포트
  기획서 : <파일명>
  생성일 : YYYY-MM-DD
  총 케이스 : N개
==============================================================

[ 섹션별 케이스 현황 ]
  섹션                      해피  언해피  엣지  합계  P0  P1  P2
  ---------------------------------------------------------------
  1. 섹션명                   n     n      n     n    n   n   n
  ...
  ---------------------------------------------------------------
  전체                        n     n      n     n    n   n   n

[ 우선순위 분포 ]
  P0: n건 (n%)  P1: n건 (n%)  P2: n건 (n%)

[ 출처 분포 ]
  기획서기반: n건 / AI추가: n건 → 기획서 보완 권고 항목

[ 커버리지 취약 섹션 ]
  엣지케이스 2개 미만이거나 P0 케이스가 없는 섹션:
  - 섹션명: 보완 권고 내용
==============================================================
```

## 설계 의도

> QA 엔지니어가 기획서 파일 하나를 입력하면 해피·언해피·엣지 케이스와 우선순위가 구조적으로 생성됩니다. 기대 결과를 실제 UI 액션 단위로 명시해 테스트 수행 시 주관적 판단 개입을 없애고, 키워드 기반 P0/P1/P2 자동 분류로 "무엇부터 막을지" 혼선을 제거합니다. 커버리지 리포트로 PM이 취약 섹션을 즉시 파악해 릴리즈 전 보완 우선순위를 결정할 수 있도록 설계했습니다.

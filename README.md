## 6. 의사결정 로그 요약

### 우선순위 판단 기준

P0·P1·P2 분류는 **서비스 영향 범위 × 사용자 이탈 가능성**을 기준으로 키워드 룩업 방식을 적용했다.

- **P0 키워드**: 앱 크래시, 강제종료, 데이터 유실, 핵심 AI 기능 완전 실패(캐릭터 생성·일기 생성), 로그인·인증 완전 차단, 계정 삭제 원자성 손상. Petbbi의 핵심 가치는 AI 생성 기능에 있으므로 캐릭터 생성·일기 생성 관련 케이스는 단일 실패만으로도 P0로 판정했다. 계정 및 인증 섹션에서 P0가 7건으로 가장 많은 이유는 인증 차단·계정 충돌·탈퇴 원자성 손상이 서비스 신뢰 자체를 위협하기 때문이다.

- **P1 판단 기준**: 기능이 작동하긴 하나 예외 처리 미구현, 오류 메시지 미노출, OS 권한 거부 미처리로 인해 사용자 플로우가 단절되거나 대안 없이 막히는 케이스. 전체 54건(69.2%)을 차지하며, 이는 기획서에 예외 흐름이 체계적으로 정의되지 않은 영역이 그만큼 많다는 것을 의미한다.

- **P2 판단 기준**: Empty State 화면 미정의, 카드 인터랙션 방식 미명시 등 서비스 이용 자체는 가능하나 UX 완성도에 영향을 주는 케이스. 6건(7.7%)으로 낮지만, 온보딩 이탈과 직결될 수 있어 출시 후 1스프린트 내 보완을 권고한다.

### 엣지케이스 발굴 로직

기획서 내용 분석 후 **5개 관점의 체계적 스위프**를 적용해 30건의 AI추가 케이스를 도출했다.

1. **권한(Permission)**: 카메라·갤러리·푸시 알림 OS 권한 거부 상태 — 모바일 앱에서 권한 거부는 최소 10~30% 사용자가 경험하는 일반적 시나리오
2. **네트워크(Network)**: AI 연산 중 연결 차단, 타임아웃 — 고비용 AI 연산이 오래 걸릴수록 네트워크 불안정과 겹칠 확률이 높아짐
3. **경계값(Boundary)**: 파일 용량 한도(10MB), 이미지 해상도(512px), 이름 최대 글자(20자), 재생성 한도(3회) — 기획서에 수치는 명시되어 있으나 한도 초과 시 UX 처리 정책이 빠져 있었음
4. **빈 상태(Empty State)**: 등록 반려동물 0마리, 팔로우 0명, 검색 결과 0건 — 신규 가입자가 가장 먼저 마주치는 화면이지만 기획서에 Empty State 디자인 명세가 전무
5. **폴백(Fallback)**: AI 서버 완전 실패, 트랜잭션 강제종료, 이메일-소셜 계정 충돌 — 정상 경로만 설계된 기획서에서 예외 복구 경로가 빠진 영역을 발굴

## [Standard] 파이프라인 사용법

> 새 기획서(.md)를 넣으면 동일한 체크리스트를 자동 생성하는 방법을 설명합니다.

### 사용 방법

**방법 1 — Claude Code 슬래시 커맨드 (권장)**

```
/gen-tc <기획서.md 경로>
```

예시:
```
/gen-tc data/petbbi-spec-v2-with-ac.md
```

Claude Code가 자동으로 아래 6단계를 순서대로 실행합니다.

| 단계 | 내용 |
|------|------|
| 1단계 구조 분석 | `##` / `###` 헤더 기준으로 섹션·서브섹션 추출 및 화면 상태·수치·AC 정의 여부 파악 |
| 2단계 케이스 설계 | Happy / Unhappy / Edge 유형으로 서브섹션별 케이스 초안 생성 |
| 3단계 우선순위 분류 | 키워드 룩업 기반 P0·P1·P2 자동 분류 |
| 4단계 엣지케이스 보완 | 권한·네트워크·경계값·빈 상태·폴백 5개 관점 체크리스트로 누락 보완 |
| 5단계 CSV 생성 | `<기획서명>_tc_YYYYMMDD.csv` 파일 자동 저장 (UTF-8 BOM) |
| 6단계 커버리지 리포트 | 섹션별 Happy·Unhappy·Edge·P0 분포 + 우선순위·출처 통계 자동 출력 |

**방법 2 — Python 스크립트 (API 키 보유 시)**

```bash
# 환경 변수 설정
$env:ANTHROPIC_API_KEY = "sk-ant-..."

# 파이프라인 실행
python tc-pipeline.py data/petbbi-spec-v2-with-ac.md

# 출력 파일 지정 시
python tc-pipeline.py data/petbbi-spec-v2-with-ac.md --output output/qa-checklist.csv
```

### 체크리스트 양식 설계 근거

이번 9컬럼 양식(`ID / 섹션 / 케이스 유형 / 테스트 시나리오 / 기대 결과 / 우선순위 / 출처 / 사전 조건 / 비고`)은 세 가지 실무 문제를 해결하기 위해 직접 설계했습니다.

**① 기획 공백 보완 — `비고` 컬럼**

기획서 분석 과정에서 "기획 미명시"로 분류된 항목들이 `비고` 컬럼에 자동 수집됩니다. 이번 분석에서만 30건의 AI추가 케이스와 함께 수십 개의 정책 공백이 발견됐습니다. `비고`는 QA가 발견한 기획 누락을 PM에게 전달하는 비동기 채널이 됩니다.

**② PM/QA 소통 비용 절감 — `출처` 컬럼 (기획서기반 / AI추가)**

`출처` 컬럼은 "이 케이스는 기획서에 근거가 있는가, 없는가"를 한 눈에 보여줍니다. PM은 `AI추가` 케이스만 필터링하면 기획 보완이 필요한 항목을 즉시 파악할 수 있습니다.

**③ 백로그 자산화 — `우선순위` + `비고` 조합**

`P0 + 비고="기획 미명시"` 케이스는 출시 전 반드시 정책을 확정해야 하는 항목입니다. 이 조합으로 필터링하면 "다음 스프린트에서 결정해야 할 설계 이슈 리스트"가 자동으로 완성됩니다.

## [Challenge] 릴리즈 게이트 & 테스트 실행 플랜

### 릴리즈 게이트 기준 (본인 설계)

| 조건 | 기준 | 판단 |
|------|------|------|
| P0 미해결 | **0건** | 1건이라도 미해결 시 출시 불가 (Hard Gate) |
| P1 미해결 | **3건 이하** | Known Issues 명시 + 2스프린트 내 해결 약속 조건으로 조건부 출시 가능 |
| P2 미해결 | 무제한 | 다음 스프린트로 이월 가능 / Empty State 관련 P2는 온보딩 지표 모니터링 병행 권고 |

> 근거: Petbbi의 핵심 차별점은 AI 캐릭터·일기 생성이므로, 해당 기능의 P0 케이스(TC-019·023·027·032·038) 미해결은 단순 버그가 아니라 제품 정체성 훼손으로 간주합니다.

### 테스트 실행 우선순위 플랜

**플랜 1 — 핵심 AI 파이프라인 검증 (출시 전 필수, 예상 소요: 1일)**

- **대상**: TC-019, TC-022, TC-023, TC-024, TC-027, TC-032, TC-038 (P0 · AI 핵심 기능 7건)
- **근거**: Petbbi의 유일한 차별점이 AI 캐릭터·일기 생성임. 해당 기능 비정상 동작 시 앱 존재 이유가 사라지며 스토어 1점 폭격으로 직결
- **실행 시점**: 빌드 완료 직후 → iOS·Android 실물 디바이스 각 1대 → 전체 통과 시 플랜 2 진행

**플랜 2 — 기획 공백 발견 AI추가 케이스 집중 검증 (출시 전 권고, 예상 소요: 0.5일)**

- **대상**: TC-013, TC-023, TC-062, TC-069, TC-077 (권한·폴백·계정 충돌 P0·P1)
- **근거**: 기획서에 정책이 없는 케이스는 개발자가 임의 처리하거나 미처리할 가능성 있음. 특히 TC-069(계정 충돌)·TC-077(탈퇴 원자성)은 발생 빈도는 낮지만 데이터 손상으로 이어지는 고위험 케이스
- **실행 시점**: 플랜 1 통과 후 → PM·개발팀 정책 확정 미팅 선행 필요

### 우선순위 점수화 방식 (본인 설계)

```
Risk Score = 영향도(1~5) × 발생 가능성(1~5) × 사용자 범위(1~3)
```

| 항목 | 기준 |
|------|------|
| 영향도 | 5: 데이터 유실·앱 크래시 / 3: 플로우 단절 / 1: UX 불편 |
| 발생 가능성 | 5: 특정 조건 없이 항상 발생 / 3: 경계값 조건에서 발생 / 1: 드문 환경 조합 |
| 사용자 범위 | 3: 전체 사용자 / 2: 특정 OS·권한 설정 사용자 / 1: 극소수 시나리오 |

| Risk Score | 우선순위 |
|:----------:|:-------:|
| 40 이상 | P0 |
| 15 ~ 39 | P1 |
| 15 미만 | P2 |

> 예시: TC-077(계정 삭제 중 강제종료) = 영향도 5 × 발생 가능성 3 × 사용자 범위 3 = **45점 → P0**  
> 예시: TC-050(피드 Empty State) = 영향도 2 × 발생 가능성 5 × 사용자 범위 1 = **10점 → P2**
